# FocusLedger 项目说明

## 1. 项目定位

FocusLedger 是一个面向中文微信公众号文章的本地研究与内容整理工具。

当前阶段的正式目标：

1. 管理公众号来源，并把文章同步到本地文章库
2. 基于文章内容、来源分组和来源标签生成日报
3. 支持通过 QClaw 在微信中调用日报
4. 逐步演进为一个基于微信公众号文章的 NotebookLM 型产品

当前产品重点已经从“自动更新凭据”转向“利用已有文章数据做知识管理、专题研究和 AI 辅助整理”。

## 2. 当前正式实现边界

### 2.1 当前保留的同步相关能力

- 来源创建
- 公众号主页链接解析
- `profile_ext` 凭据保存
- 凭据状态检测
- 凭据手动更新
- 后台同步任务
- 文章去重、正文解析、摘要和标签

### 2.2 当前不再作为主流程的能力

以下能力已经退出正式主链路：

- 自动启动 Fiddler
- 自动读取抓包结果
- 自动刷新凭据
- Fiddler callback / capture 主流程

当前正式流程是：

1. 用户发起同步
2. 系统先验证凭据
3. 凭据有效则继续同步
4. 凭据失效则明确失败，并提示用户手动更新凭据
5. 用户在 `/collect` 页面粘贴新的 `profile_ext` 链接
6. 再重新发起同步

### 2.3 重要现实约束

- 微信公众号凭据具有时效性
- 当前版本无法保证“完全无人工参与”的最新文章获取
- 因此产品价值不再建立在自动获取最新文章之上

## 3. 当前产品方向

当前阶段 FocusLedger 更接近：

**一个基于微信公众号文章数据的本地知识库、工作区和 AI 辅助整理系统**

重点方向包括：

- 收藏管理
- 标签管理
- 工作区 / Notebook
- 基于工作区的问答与内容生成
- QClaw 触发与结果回传
- 后续通过 MCP 暴露 FocusLedger 能力

产品方向整理见：

- `PRODUCT_NOTEBOOKLM_DIRECTION.md`

## 6. 已确认的产品边界

### 6.1 凭据不是永久有效

来源创建成功不代表后续永远可用。  
同步前必须显式验证凭据，凭据失效时必须明确返回失败和提示，不能误报成功。

### 6.2 当前同步链路允许人工参与

当前版本接受这样一个事实：

- 用户需要手动更新过期凭据
- 系统负责检测、提示、校验和继续同步

### 6.3 当前主价值不在“自动抓最新”

当前最值得投入的部分是：

- 已有文章数据的组织
- 文章库管理
- 工作区构建
- AI 助理交互

## 7. 已放弃或降级的技术路径

这些路径不再作为主线投入：

### 7.1 mitmproxy

结论：

- 当前环境下不稳定
- 无法稳定复现手工链路效果

### 7.2 自研 Windows helper

技术路线：

- `.NET 8 + Titanium.Web.Proxy`

结论：

- 理论可行
- 真实微信 PC 环境里不够稳定
- 不再作为主方案

### 7.3 FiddlerCore

结论：

- 商业依赖
- 管理复杂
- 不适合作为当前仓库主实现方向

### 7.4 Fiddler Classic 半自动刷新主流程

结论：

- 实现和维护成本过高
- 稳定性不足
- 不再作为正式产品主流程

## 9. 协作与清理建议

### 建议纳入 Git

- 前端代码
- 后端代码
- README
- AGENTS.md
- 数据库迁移文件
- 产品文档
- QClaw 技能定义
- 启动脚本

### 不应纳入 Git

- `.env`
- 本地日志
- 本地缓存
- 临时导出文件
- `_ref_Access_wechat_article/`
- `.claude/`

### 分支建议

- `main`：保持可运行
- 功能开发从 `main` 拉分支
- 合并前至少确认：
  - 前端可启动
  - 后端可启动
  - 关键页面可访问
  - 当前正式同步链路未被破坏

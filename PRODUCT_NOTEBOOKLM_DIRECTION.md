# FocusLedger 新产品方向整理

## 1. 方向调整背景

FocusLedger 之前长期投入的重点，是围绕微信公众号文章抓取链路中的“凭据自动刷新”能力展开。

但基于真实开发、反复测试和近期决策，当前已经可以得出一个明确判断：

- 微信公众号文章获取链路中的凭据具有明显时效性
- 自动刷新链路高度依赖微信 PC、代理抓包、Windows 环境与具体时序
- 这条链路实现成本高、稳定性差，不适合作为当前阶段的核心产品价值

因此，当前产品重点不再放在“自动化获取最新公众号文章”，而是转向：

**充分挖掘已经获取到的公众号文章数据价值，把 FocusLedger 做成一个面向微信公众号文章内容的 NotebookLM 型产品。**

这意味着，产品主线从“采集最新内容”转向“管理、组织、理解、利用已有内容”。

---

## 2. 新的产品定位

FocusLedger 的新定位可以概括为：

**一个基于微信公众号文章数据的本地知识管理、专题研究与 AI 助理系统。**

它的核心不再是“最新抓取能力”，而是：

- 对已有公众号文章进行系统化管理
- 帮助用户围绕特定主题建立研究工作区
- 基于文章集合进行问答、总结、报告生成与内容再生产

如果用类比来理解：

- 在产品形态上，接近 Google NotebookLM
- 在数据来源上，聚焦微信公众号文章
- 在用户心智上，服务于“微信收藏积灰”和“公众号内容研究整理”

---

## 3. 核心用户问题

### 3.1 “收藏等于学习”的伪学习问题

很多用户在微信使用中都会遇到同一种情况：

- 看到一篇好文章就先收藏
- 当下没有时间仔细阅读
- 之后也很少回头系统整理
- 长期积累出大量“积灰收藏”

这些内容本身有价值，但缺少一个真正适合后续利用的管理与理解系统。

### 3.2 微信公众号文章缺少结构化管理方式

公众号文章通常具有以下特点：

- 来源多
- 主题杂
- 内容长
- 时间跨度大
- 微信原生收藏和阅读机制不适合做深度检索和复用

用户真正需要的是：

- 更灵活的归档方式
- 更强的标签系统
- 更适合研究与复盘的组织方式

### 3.3 缺少面向“专题研究”的内容工作区

当用户想围绕某个主题做研究时，例如：

- AI Agent
- 某家公司分析
- 某个政策方向跟踪
- 某一类投资逻辑梳理

他们需要把相关文章聚集到一个地方，形成专题资料池，再围绕这个资料池做问答、写作和整理。

这正是 NotebookLM 类产品的价值所在。

---

## 4. 新产品的核心理念

### 4.1 公众号文章不是“信息流”，而是“知识资产”

FocusLedger 的目标不是只让用户浏览文章，而是把公众号文章沉淀为可以管理、检索、组合和再利用的知识资产。

### 4.2 收藏不应止于存放，而应进入整理与利用

系统的目标不是替用户堆积文章，而是帮助用户完成：

- 收藏
- 归档
- 打标签
- 建立专题工作区
- 基于内容开展 AI 问答与生成

### 4.3 AI 不是单次总结器，而是知识助理

AI 不只负责对单篇文章做摘要，还应围绕整套文章数据库和专题工作区发挥作用，例如：

- 帮助检索
- 帮助分类
- 帮助打标签
- 帮助建立专题集合
- 帮助生成研究输出

---

## 5. 产品形态设想

### 5.1 公众号文章收藏与管理系统

这是最基础、也最重要的一层。

目标是构建一个真正可用的公众号文章数据库管理系统，而不是简单的文章列表页。

应重点强化的能力包括：

- 收藏文章
- 取消收藏
- 手动归档
- 多维标签
- 来源管理
- 按时间、来源、标签、主题等维度筛选
- 更强的搜索与排序

它可以理解为：

**一个面向公众号文章的“收藏库 + 文章资产库”。**

### 5.2 工作区 / Notebook 系统

这是产品从“文章管理工具”走向“研究工作台”的关键一步。

可以设计为：

- 用户创建一个工作区 / Notebook
- 为工作区设置主题
- 将相关文章加入该工作区
- 让工作区成为一个可对话、可生成内容、可持续整理的专题知识空间

工作区中的文章来源可以是：

- 从文章库手动挑选
- 按标签批量加入
- 按来源批量加入
- 按搜索结果批量加入
- 后续由 AI 推荐加入

工作区的价值在于：

- 把零散文章组织成专题
- 把“收藏”转化为“研究资料”
- 让 AI 的理解边界从单篇扩展到一组来源

### 5.3 基于工作区的 AI 能力

当工作区建立后，AI 才真正从“单篇摘要器”升级为“研究助理”。

围绕工作区，未来可支持的 AI 能力包括：

- 工作区问答
- 关键观点提取
- 多篇文章交叉总结
- 主题脉络梳理
- 争议点与分歧点整理
- 时间线整理
- 研究报告生成
- 博客文章生成
- PPT 文案生成
- 专题 briefing 生成

也就是说，工作区不只是“文章容器”，还是 AI 的上下文边界。

### 5.4 AI 助理式管理系统

除了让用户手动管理文章与标签，还希望增加一套“AI 助理式管理系统”，通过自然语言帮助用户完成日常管理工作。

例如用户可以对 AI 说：

- 帮我把最近一周所有关于 AI Agent 的文章收藏起来
- 给这些文章打上“行业趋势”和“产品观察”标签
- 创建一个 “AI Agent 研究” 工作区，并把相关文章加进去
- 找出我收藏里关于 OpenAI、Anthropic、Google 的文章
- 帮我整理过去一个月收藏中最值得深读的十篇

这意味着 AI 不只是回答问题，还要参与：

- 收藏管理
- 标签管理
- 工作区管理
- 内容筛选与归类

对应的产品特征是：

- 用户可以手动操作
- 用户也可以通过自然语言驱动系统完成操作
- AI 成为文章数据库与知识空间的管理助理

## 6. AI 助理系统的实现思路

这是当前需要单独明确的一层，因为它不仅是功能问题，也是成本和架构问题。

### 6.1 AI 助理的目标不是单纯对话，而是“可执行管理”

这套 AI 助理系统的目标不是做一个聊天框，而是让用户通过自然语言完成真实操作，例如：

- 收藏一批文章
- 修改一批标签
- 建立一个新工作区
- 生成某个主题的文章清单
- 基于某个工作区生成报告、博客、PPT、播客脚本

所以它本质上是：

**自然语言入口 + 结构化能力调用 + 内容生成系统**

### 6.2 为什么会考虑 `QClaw -> Codex CLI -> MCP -> FocusLedger`

这个思路的核心原因不是技术炫技，而是现实约束：

- 如果在 FocusLedger 产品内直接接入 LLM API，需要额外承担 API token 成本
- Codex CLI 已经订阅，可直接使用，不需要再为同类能力单独付费
- QClaw 的价值在于接入微信，但其内置模型能力有限且有额度限制
- 因此，从成本和效果两方面看，让 QClaw 把复杂任务交给 Codex CLI，是一个很自然的想法

换句话说，这条链路的动机是：

**尽量复用已经订阅的 AI 能力，而不是为产品内再单独采购一套模型调用成本。**

### 6.3 这个思路的优点

如果这条链路可稳定落地，它的优势很明确：

- 降低额外 API 成本
- 充分利用 Codex CLI 的能力上限
- 让 QClaw 继续承担微信入口角色
- 让 FocusLedger 继续专注于数据和业务能力

从用户视角，这会形成一种很有吸引力的体验：

- 用户在微信里给 QClaw 发请求
- QClaw 把任务交给本地 AI 工具处理
- AI 调用 FocusLedger 的数据与管理能力
- 最终把文本、Markdown、清单、报告或音频文件回传到微信手机端

### 6.4 这条链路的风险

虽然这条思路在成本上很有吸引力，但它也有明显风险：

- 链路过长，依赖多个系统协同
- 运行稳定性取决于 QClaw、Codex CLI、MCP 调用链是否都可靠
- 用户环境配置复杂，部署门槛高
- 一旦任一环节失效，问题定位会比较困难

因此，它更像是：

- 一个高价值的低成本架构方向
- 一个适合本地个人工作流的强方案
- 但不一定适合作为唯一主链路的产品 runtime 架构

### 6.5 当前更合理的架构判断

基于现阶段目标，我建议把这件事分成两层看：

#### 第一层：FocusLedger 的产品能力层

FocusLedger 本身应负责：

- 文章数据库
- 收藏与标签系统
- 工作区系统
- 检索、筛选和结构化能力
- 报告 / 博客 / PPT / 播客脚本等生成任务

也就是说，FocusLedger 应是“知识库与能力引擎”。

#### 第二层：AI 调用入口层

AI 调用入口可以有多种：

- 产品内直接接 LLM API
- 通过 MCP 暴露给 Codex CLI 等外部 AI 工具
- 由 QClaw 作为微信端入口来触发

因此，更合理的分工不是单选题，而是：

- 产品内 AI：未来可评估是否接 LLM API
- 产品外 AI：优先考虑 MCP
- 微信入口：由 QClaw 承担
- 高能力执行器：Codex CLI 可作为重要外部调用方

一句话概括：

**产品内可以暂不重度依赖 API，产品外优先通过 MCP 接入已订阅的 AI 能力。**

### 6.6 当前阶段的务实结论

从“课程作业 + 成本控制 + 现有订阅条件”来看，当前阶段最务实的路线是：

1. FocusLedger 先把“知识库、收藏、标签、工作区、生成任务”能力做扎实
2. 为 FocusLedger 设计可被调用的结构化接口
3. 优先考虑把这些接口整理为 MCP 能力
4. 让 Codex CLI 成为一个重要的外部 AI 调用入口
5. 让 QClaw 作为微信端触发和结果回传通道

这意味着，现阶段不必急着在产品代码里直接绑定 LLM API。

## 7. 可预期的系统角色分工

如果沿着上述方向推进，未来系统中几个关键角色可以这样定位：

### 7.1 FocusLedger

定位：

- 本地文章知识库
- 收藏管理系统
- 标签管理系统
- 工作区 / Notebook 系统
- 内容生成任务引擎

### 7.2 MCP

定位：

- FocusLedger 的外部标准化能力接口
- 供 Codex CLI 等 AI 工具读取和操作 FocusLedger

适合暴露的能力例如：

- 搜索文章
- 查询标签
- 批量收藏 / 取消收藏
- 批量打标签
- 创建工作区
- 向工作区添加文章
- 基于工作区生成报告或清单

### 7.3 Codex CLI

定位：

- 一个高能力、低额外成本的 AI 调用端
- 重点承担复杂分析、自然语言理解与任务编排

它不是 FocusLedger 的唯一入口，但在当前成本约束下，可以成为最重要的外部智能入口之一。

### 7.4 QClaw

定位：

- 微信端入口
- 任务触发层
- 结果回传层

尤其在你已经确认 QClaw 支持发送文件到微信手机端后，它的价值会进一步放大。

它不仅能回传文本，还能回传：

- Markdown 报告
- 文章清单
- 博客草稿
- PPT 文案
- 播客音频文件

## 8. 可实现的交互场景

### 8.1 微信中请求专题文章清单

用户在微信里说：

- 帮我整理最近一个月关于 AI Agent 的公众号文章

系统链路可以是：

- QClaw 接收请求
- Codex CLI 理解任务并调用 FocusLedger
- FocusLedger 返回文章集合
- QClaw 把整理后的清单发回微信

### 8.2 微信中请求专题报告

用户在微信里说：

- 基于我“AI Agent 研究”工作区生成一份简报

系统返回：

- 摘要版文本
- Markdown 文件
- 后续可扩展为 PDF 或演示材料

### 8.3 微信中请求文章管理动作

用户在微信里说：

- 帮我把最近收藏里关于 OpenAI 和 Anthropic 的文章打上“大模型公司”标签

这类能力体现的正是 AI 助理式管理，而不是单纯内容问答。

### 8.4 微信中请求播客内容生成

用户在微信里说：

- 基于这个工作区生成一期播客脚本，并导出音频

理想链路可以是：

- Codex CLI 负责文本组织与脚本生成
- FocusLedger 负责提供工作区内容
- 后续接入 TTS 或音频生成环节
- QClaw 把最终音频文件发送到微信手机端

这类场景对于课程展示也很有吸引力，因为它体现了：

- 本地知识库
- AI 组织能力
- 微信端交互
- 文件型结果交付

## 9. 与当前项目资产的关系

这个新方向并不是推翻现有项目，而是重新利用已经积累的核心资产。

当前项目已经具备的有价值基础包括：

- 公众号来源管理
- 文章数据库
- 标签与分组基础
- 正文清洗
- 摘要与结构化能力
- 日报生成能力
- 本地化部署架构
- QClaw 集成经验

这些能力都可以成为新方向的基础设施。

尤其是：

- 文章库是 Notebook 的素材层
- 标签与来源系统是知识组织层
- 生成能力是未来 AI 助理层的基础
- QClaw 是未来微信端入口

换句话说，FocusLedger 并没有失去价值，而是应该把价值重心从“采集最新数据”转移到“激活已有数据”。

---

## 10. 课程作业场景下的价值表达

在课程作业语境下，这个方向也更容易形成完整闭环。

相比“自动刷新凭据并获取最新公众号文章”，新的讲法更清晰：

- 面向真实用户习惯提出问题：“微信收藏等于学习，但内容长期积灰”
- 基于公众号文章构建本地知识资产系统
- 通过收藏、标签和工作区完成内容组织
- 通过 AI 对工作区进行问答、整理和内容生成
- 通过 QClaw 实现微信端触发和结果回传

这条叙事更稳定，也更容易展示产品价值。

---

## 11. 当前阶段的开发建议

在新方向下，下一阶段开发建议优先围绕以下目标展开：

1. 强化文章数据库与收藏管理能力
2. 建立更完整的标签系统
3. 设计并实现工作区 / Notebook 机制
4. 明确 FocusLedger 内部“可执行能力”的接口边界
5. 评估并设计适合 FocusLedger 的 MCP 接口
6. 让 QClaw + Codex CLI + FocusLedger 形成最小可用闭环

而“凭据自动刷新”则应降级为：

- 非当前阶段主线
- 非核心卖点
- 不再决定产品成败

---

## 12. 一句话总结

**FocusLedger 的下一阶段，不再把主要精力投入到高不确定性的公众号最新文章自动获取，而是把已有公众号文章数据做成一个支持收藏管理、专题研究、AI 助理操作和微信端调用的 NotebookLM 型产品；在成本受限的前提下，优先考虑通过 MCP 复用已订阅的 Codex CLI 能力，而不是立即在产品内直接接入额外付费的 LLM API。**

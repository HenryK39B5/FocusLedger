# FocusLedger 产品方向（NotebookLM 路线）

## 1. 当前定位

FocusLedger 是一个运行在本地的微信公众号文章研究平台。

当前产品重心已经明确从“自动抓取最新文章”转向“管理、组织、理解、利用已有文章资产”。核心能力包括：

- 本地文章库管理
- 来源管理与手动凭据维护
- 标签、收藏、LLM 总结
- Notebook 工作区
- 基于 Notebook 的 AI 对话、播客脚本与音频生成
- 面向外部 AI agent 的任务型协作接口

当前不再作为主线投入的方向：

- 自动刷新公众号凭据
- Fiddler / mitmproxy 抓包自动化
- 日报链路

---

## 2. 产品目标

FocusLedger 要做成一个面向微信公众号文章的本地版 NotebookLM。

它解决的不是“如何持续自动抓到最新文章”，而是：

- 如何把已经获得的公众号文章管理好
- 如何把零散收藏变成结构化研究资料
- 如何围绕一个主题快速建立 Notebook 工作区
- 如何让 AI 基于这些资料完成问答、整理与内容生成

一句话概括：

**FocusLedger 是一个面向微信公众号文章的本地研究与知识工作台。**

---

## 3. 核心使用场景

### 3.1 已有文章资产管理

用户已经通过历史同步、手动收集或链接导入拿到一批公众号文章，希望：

- 搜索文章
- 收藏文章
- 编辑标签
- 查看来源
- 批量做 LLM 总结
- 将文章组织进 Notebook

### 3.2 面向主题的 Notebook 研究

用户围绕某个主题建立 Notebook，例如：

- AI Agent
- 宏观策略
- 某家公司研究
- 某个政策主题

然后把相关文章加入 Notebook，在 Notebook 中进行：

- AI 对话
- 播客脚本生成
- 音频生成
- 后续可扩展的报告 / 思维导图 / 博客生成

### 3.3 微信端轻量采集

一个高价值场景是：

- 用户在手机微信里看到一篇或几篇不错的公众号文章
- 用户把链接发给某个 AI agent 工具
- 该工具调用 FocusLedger
- FocusLedger 把文章直接加入本地文章库

这个场景不依赖自动刷新凭据，但非常符合真实使用习惯，也适合课程作业展示。

---

## 4. AI Agent 集成方向

此前曾设想过链路：

- `AI agent 工具 -> Codex CLI -> MCP -> FocusLedger`

这个思路可以复用既有模型能力，但链路过长，稳定性与故障定位都不理想。

当前正式方向已经收敛为：

- `AI agent 工具 -> FocusLedger`

也就是说：

- 外部 agent 工具承接用户请求
- FocusLedger 直接提供结构化能力
- Codex CLI 不再作为正式调用链路中的必经节点

这样做的原因：

- 降低链路复杂度
- 降低不稳定性
- 更适合课程作业展示
- 更容易形成可维护、可解释的产品结构

---

## 5. 来源与文章入库策略

为了让外部 agent 真正可用，FocusLedger 需要一项基础能力：

**支持输入单篇或多篇公众号文章链接，直接抓取文章并入库。**

这项能力应具备：

- 支持单篇链接
- 支持多篇链接
- 直接抓取正文并入库
- 自动识别文章对应公众号
- 如来源不存在，则自动创建来源

### 5.1 来源允许“无凭据创建”

为支持上面的轻量采集场景，来源模型必须进一步明确：

- “来源存在”不等于“已具备有效凭据”

因此后续约定应当是：

- 添加来源时允许凭据为空
- 自动创建来源时默认凭据为空
- 只有在需要历史同步该来源时，才要求补全凭据

这样可以把“来源管理”和“同步能力”解耦。

## 6. 开发顺序

当前建议分两步推进：

1. 先把 FocusLedger 原生能力做稳
2. 再开发一套较完整的 `agent -> FocusLedger` skill

### 阶段 1：打通 FocusLedger 原生能力

目标：

- 支持单篇 / 多篇文章链接直入库
- 自动创建对应来源
- 来源允许无凭据创建
- 提供一个前端最小测试入口

阶段 1 的重点不是某个外部调用端，而是先把 FocusLedger 自身能力做稳。

### 阶段 2：开发较完整的 `agent -> FocusLedger` skill

这不是做一个极窄的“最小可用 skill”，而是做一组边界清晰、围绕真实任务场景设计的能力。

目标：

- 让用户在手机微信或其他 AI 工具界面中远程操作 FocusLedger
- 覆盖文章采集、文章库整理、Notebook 管理与 Notebook 输出
- 在课程作业展示中体现完整产品价值，而不是只展示“导入一篇文章”

#### 2.1 Skill 的能力边界

建议将正式能力边界划分为四组：

##### A. 文章采集与入库

- 导入单篇 / 多篇公众号文章链接
- 返回导入结果摘要
- 自动创建来源
- 查询某篇文章是否已在库中

##### B. 文章库检索与整理

- 搜索文章
- 按来源 / 标签 / 时间范围筛选文章
- 收藏 / 取消收藏文章
- 为文章增加 / 删除标签
- 对一篇或多篇文章触发 LLM 总结

##### C. Notebook 工作区管理

- 创建 Notebook
- 更新 Notebook 元信息
- 查询 Notebook 列表
- 将文章加入指定 Notebook
- 查看某个 Notebook 当前有哪些文章

##### D. Notebook 研究输出

- 基于 Notebook 发起 AI 对话
- 基于 Notebook 生成播客脚本
- 基于 Notebook 生成音频
- 返回脚本、状态或文件结果

#### 2.2 Skill 设计原则

不要把 skill 设计成“远程 CRUD 面板”，而是设计成“任务型能力集合”。

更适合暴露的动作是：

- `import_articles_from_links`
- `search_articles`
- `update_article_tags`
- `summarize_articles`
- `create_notebook`
- `add_articles_to_notebook`
- `ask_notebook`
- `generate_notebook_podcast_script`
- `generate_notebook_audio`

这样做的好处：

- 更符合用户心智
- skill 文档更容易维护
- 外部 agent 更容易稳定调用
- 内部实现调整时，对外边界更稳

## 7. HTTP API 与 MCP 的优先级

当前阶段不建议先做 MCP。

更合理的顺序是：

1. 先做清晰、稳定的 HTTP API
2. 让外部 agent 先通过 skill 调用这些 API
3. 如有必要，再把这些能力抽象为 MCP

原因：

- FastAPI 现成可用
- 本地调试最简单
- 错误定位更直接
- 当前阶段更重视稳定演示，而不是额外抽象层

所以当前技术建议是：

**先 API，后 MCP。**

---

## 8. 当前阶段的产品判断

当前最值得投入的，不是继续攻坚“自动更新凭据”，而是把下面这条链路做扎实：

- 文章资产管理
- Notebook 研究工作区
- 文章链接直入库
- 外部 agent 调用 FocusLedger

这条路径更稳定、更符合真实使用习惯，也更能体现 FocusLedger 的产品价值。

---

## 9. 一句话总结

**FocusLedger 的下一阶段主线，是把文章链接直入库、来源无凭据创建、Notebook 研究工作区与面向外部 AI agent 的任务型 skill 做成一条稳定、清晰、可演示的本地研究工作流。**

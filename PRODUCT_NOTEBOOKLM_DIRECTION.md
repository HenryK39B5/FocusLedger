# FocusLedger 产品方向（NotebookLM 路线）

## 1. 当前定位

FocusLedger 是一个运行在本地的微信公众号文章研究平台。

当前产品重心已经明确从“自动抓取最新文章”转向“管理、组织、理解、利用已有文章资产”。

核心能力包括：

- 本地文章库管理
- 来源管理与手动凭据维护
- 标签、收藏、LLM 总结
- Notebook 工作区
- 基于 Notebook 的 AI 对话、播客脚本与音频生成

当前不再作为主线投入的方向：

- 自动刷新公众号凭据
- Fiddler / mitmproxy 抓包自动化
- 日报链路

---

## 2. 产品目标

FocusLedger 要做成一个面向微信公众号文章的本地版 NotebookLM。

它解决的不是“如何持续自动抓到最新文章”，而是：

- 如何把已经获取到的公众号文章管理好
- 如何把零散收藏变成结构化研究资料
- 如何围绕一个主题快速建立 Notebook 工作区
- 如何让 AI 基于文章资产帮助用户完成问答、整理与内容生成

一句话概括：

**FocusLedger 是一个面向微信公众号文章的本地研究与知识工作台。**

---

## 3. 核心使用场景

### 3.1 已有文章资产管理

用户已经通过历史同步、手动收集或后续导入拿到一批公众号文章，希望：

- 搜索文章
- 收藏文章
- 编辑标签
- 查看来源
- 批量做 LLM 总结
- 将文章组织进 Notebook

### 3.2 面向主题的 Notebook 研究

用户围绕某个主题建立 Notebook，例如：

- AI Agent
- 宏观策略
- 某家公司研究
- 某个政策主题

然后把相关文章加入 Notebook，在 Notebook 中进行：

- AI 对话
- 播客脚本生成
- 音频生成
- 后续可扩展的报告 / 思维导图 / 博客生成

### 3.3 微信端轻量采集

一个很重要的新场景是：

- 用户在手机微信里看到一篇或几篇不错的公众号文章
- 用户把链接发给 QClaw
- QClaw 调用 FocusLedger
- FocusLedger 把这些文章直接加入文章库

这个场景不依赖“自动刷新凭据”，却非常符合真实使用习惯，也更适合课程作业演示。

---

## 4. QClaw 集成方向调整

此前曾设想过链路：

- `QClaw -> Codex CLI -> MCP -> FocusLedger`

这个思路的优点是可以复用已订阅的 AI 能力，但链路过长，运行稳定性和故障定位都比较差。

当前建议正式调整为：

- `QClaw -> FocusLedger`

也就是说：

- QClaw 负责承接微信端请求
- FocusLedger 直接提供结构化能力
- Codex CLI 不再作为正式调用链路中的必经节点

这样做的原因：

- 降低链路复杂度
- 降低运行不稳定性
- 更适合课程作业展示
- 更容易形成可维护、可解释的产品架构

---

## 5. 新增能力方向：文章链接直入库

为了让 QClaw 真正可用，FocusLedger 需要新增一项基础能力：

**支持输入单篇或多篇公众号文章链接，直接抓取文章并入库。**

这项能力的价值很高，因为它可以成为：

- 用户手动导入文章的入口
- QClaw skill 的第一项核心能力
- 后续 Notebook 工作流的上游输入口

这项能力应当具备以下特征：

- 支持单篇链接
- 支持多篇链接
- 直接抓取正文并入库
- 自动识别文章来源公众号
- 如果来源不存在，则自动创建来源

### 5.1 来源允许“无凭据创建”

为支持这个场景，来源模型需要进一步明确：

- “来源存在”不等于“已经具备有效凭据”

也就是说，来源应分为两种状态：

1. 仅用于文章归档与管理的来源
2. 具备有效 `profile_ext`、可执行历史同步的来源

因此后续产品约定应当是：

- 添加来源时，允许凭据为空
- 自动创建来源时，默认凭据为空
- 后续需要同步该来源时，再手动补全凭据

这样既方便管理，又不会把“来源管理”和“同步能力”强耦合。

## 6. 下一阶段开发顺序

下一阶段建议分三步推进，而不是一次性同时做 QClaw、MCP、自然语言代理和复杂工作流。

### 阶段 1：打通 FocusLedger 原生能力

目标：

- 支持单篇 / 多篇文章链接直接入库
- 自动创建对应来源
- 来源允许无凭据创建
- 提供一个前端最小测试入口

阶段 1 的重点不是 QClaw，而是先把 FocusLedger 自身能力做稳。

原因：

- 如果 FocusLedger 本身还不能稳定处理链接入库，QClaw 接入只会放大问题
- 先做成原生能力，后续无论接 skill 还是 MCP 都更简单

### 阶段 2：开发 QClaw 最小可用 skill

在阶段 1 稳定后，再做 QClaw 接入。

第一版 skill 聚焦一个最清晰、最真实的场景：

- 用户发送一个或多个公众号文章链接
- QClaw 调用 FocusLedger 的文章导入接口
- QClaw 返回导入结果摘要

返回结果可以包括：

- 成功导入多少篇
- 新建了多少个来源
- 哪些链接已存在
- 哪些链接导入失败

这会形成一个很强的 MVP：

**手机微信 -> QClaw -> FocusLedger 入库**

### 阶段 3：扩展 QClaw 对 FocusLedger 的操作面

在前两阶段跑通后，再逐步扩展到更完整的远程研究操作。

后续可扩展能力包括：

- 创建 Notebook
- 将文章加入指定 Notebook
- 查询 Notebook 中的来源文章
- 触发某个 Notebook 的 AI 对话或脚本生成
- 触发文章批量 LLM 总结
- 让 QClaw 返回报告、脚本或音频文件

---

## 7. HTTP API 与 MCP 的优先级判断

这一阶段不建议先做 MCP。

更合理的顺序是：

1. 先做清晰、稳定的 HTTP API
2. 让 QClaw 先通过 skill 调用这些 API
3. 后续如有必要，再把这些能力抽象为 MCP

原因：

- FastAPI 现成可用
- 本地调试最简单
- 错误定位更直接
- 课程作业阶段优先追求稳定可演示

所以当前技术建议是：

- **先 API，后 MCP**

---

## 8. 当前阶段的产品判断

当前最值得投入的，不是继续攻坚“自动更新凭据”，而是把下面这条链路做扎实：

- 文章资产管理
- Notebook 研究工作区
- 文章链接直入库
- QClaw 远程调用 FocusLedger

这条路径更稳定、更符合真实使用习惯，也更能体现 FocusLedger 的产品价值。

---

## 9. 一句话总结

**FocusLedger 的下一阶段主线，是先把“文章链接直入库 + 无凭据来源创建”做成稳定原生能力，再基于这个能力开发 QClaw 直连 FocusLedger 的最小可用 skill，最后逐步扩展到 Notebook 工作流的远程调用。**
